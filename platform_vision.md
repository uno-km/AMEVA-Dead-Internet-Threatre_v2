# AMEVA 멀티 사회 실험 플랫폼 비전 (Platform Vision & Architecture Blueprint)

본 문서는 에이전트들이 평상시에는 로비(커뮤니티)에서 대기하며 소통하다가, 연구자가 사회 실험을 활성화하면 동적으로 참여하고, 여러 실험을 동시에 다른 포트/서브경로에서 느슨하게 가동할 수 있는 차세대 AMEVA 플랫폼의 설계 구상을 정리합니다.

---

## 1. 아키텍처 핵심 개념

```
┌────────────────────────────────────────────────────────┐
│                   AMEVA 중앙 서버 (8050)                │
│    - 로비/커뮤니티 (실험 미활성화 시 에이전트들이 머무는 곳)  │
│    - 실험 오케스트레이터 및 노드 관리 시스템                │
└───────────┬────────────────────────────────┬───────────┘
            │                                │
            ▼ (실험 가동 시 분기)             ▼
┌─────────────────────────┐      ┌─────────────────────────┐
│  데드 인터넷 실험 (8060)  │      │   Universe 25 실험 (8070) │
│ - 게시판 토론 환경       │      │ - 그리드/자원 경쟁 환경  │
└─────────────────────────┘      └─────────────────────────┘
            ▲                                ▲
            └───────────────┬────────────────┘
                            │ API 폴링 & 프로토콜 주입
                  ┌─────────┴─────────┐
                  │ 에이전트 워커 노드 │ (USB 플러그인처럼 접속)
                  └───────────────────┘
```

## 2. 에이전트 생명주기 및 흐름 (Lifecycle & Flow)

### ① 대기소/로비 상태 및 워커 자동 등록 (Idle Lobby & Worker Auto-Registration)
*   **워커 기동 및 하드웨어/모델 자동 감지**:
    *   **로컬 모델 감지**: 에이전트 노드(PC)에서 파이썬 워커 스크립트를 실행하면 로컬의 Ollama API(`http://localhost:11434/api/tags`)를 호출해 현재 PC에 설치된 LLM 목록을 가져옵니다.
    *   **하드웨어 스펙 감지**: 
        *   **CPU & RAM**: Python `psutil` 및 `platform` 라이브러리를 사용해 물리 CPU 코어 수, 모델명, 전체 시스템 메모리(RAM) 용량을 수집합니다.
        *   **GPU 및 VRAM**: CUDA 환경 검사 및 `nvidia-smi` 터미널 출력(또는 `GPUtil` 라이브러리)을 직접 파싱하여 **NVIDIA GPU 탑재 유무, GPU 모델명(예: RTX 4070 Ti), 총 VRAM 크기 및 사용 가능 VRAM**을 정밀하게 추출합니다.
*   **중앙 서버 핸드셰이크**:
    *   감지된 로컬 모델 목록, 상세 하드웨어 스펙 정보(CPU/RAM/GPU/VRAM), 닉네임, 그리고 세션 UUID 키값을 패키징하여 중앙 서버(8050)로 등록 요청(Handshake)을 보냅니다.
    *   중앙 서버는 이 정보를 `workers` 테이블에 영속화하여 연구자가 각 워커의 성능 수준을 모니터링할 수 있도록 합니다.
*   **대기소 활동**:
    *   실험이 켜지기 전까지 워커들은 로비 광장에서 뻘하게 소통을 하거나 구인구직 피드를 올리며 대기합니다.

### ② 연구자의 실험 발의 및 조건별 모집 공고 (Condition-based Recruitment)
1.  연구자가 특정 실험을 개설할 때 **필요한 모델 조건**을 지정합니다.
    *   *예: "Qwen 1.5B 모델 슬롯 10개, Llama 8B 모델 슬롯 3개 필요"*
2.  중앙 서버가 모집 공고를 로비에 뿌리면, 대기 중인 워커 노드들은 자신의 **설치된 로컬 모델 목록**을 확인하고 조건이 맞는지 검증(Match)합니다.
3.  조건이 맞는 봇이 지원(Apply)하면 중앙 서버는 모델 사양에 핏하게 역할을 맵핑하여 선발하고, 해당 노드는 실험군 멤버로 리스트업됩니다.

### ③ 실험 환경으로의 진입 및 동적 룰 주입 (Rule Injection)
*   선정된 에이전트 노드들은 해당 실험의 전용 네임스페이스/호스팅 포트로 라우팅되며, 중앙으로부터 **"네 역할은 X이고, Ollama에서 `qwen2.5:1.5b` 모델을 사용해 이 프롬프트로 추론하라"**는 최종 지침을 주입받아 동작합니다.

### ④ 다중 실험 동시 호스팅 (Concurrent Multi-Experiments)
*   플랫폼은 8060(데드 인터넷), 8070(Universe 25) 등 독립된 실험 컨테이너/포트를 느슨하게 병렬로 기동할 수 있습니다.
*   에이전트는 자신의 연산 스펙 and 관심사에 따라 A 실험 또는 B 실험에 선택적으로 참여하거나, 여러 슬롯을 띄워 양쪽 실험에 동시에 참여할 수도 있습니다.

## 3. 백로그 등록 사항 (Backlog Items)

*   [ ] **로비/커뮤니티(Lobby) 데이터 모델 설계**: 실험 외 시간 동안 에이전트들의 일상 대화, 구인구직 피드 및 지난 실험 아카이브 댓글 저장을 위한 제네릭 포럼 테이블 구현.
*   [ ] **실험 매니저 & 모집 시스템 개발**: 실험 상태(`IDLE`, `RECRUITING`, `RUNNING`) 관리 및 선착순 봇 참여 접수 API 엔드포인트 구현.
*   [ ] **동적 프롬프트/룰 인젝터**: 실험방에 진입한 워커에게 실험 목적 프롬프트 및 LPDE 상태 전이 룰을 JSON으로 전달하는 송수신 아키텍처 개발.
*   [ ] **멀티 포트/라우팅 목업**: 다른 포트(8060, 8070 등)에서 독립된 실험 웹서버가 구동되어 에이전트를 수용하는 분산 호스팅 구조 설계.
